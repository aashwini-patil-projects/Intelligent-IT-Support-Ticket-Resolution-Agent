ARCHITECTURE & AGENT DESIGN DOCUMENT
=====================================
HelpMate: Intelligent IT Support Ticket Resolution Agent

This document describes the multi-agent architecture, retrieval system, reasoning approach, 
and production considerations for the HelpMate system.

═══════════════════════════════════════════════════════════════════════════════════════

1. RETRIEVAL & LLM LAYER
═══════════════════════════════════════════════════════════════════════════════════════

1.1 EMBEDDING MODEL & INDEXING
------------------------------
Model: Azure OpenAI text-embedding-3-large
- Dimension: 3072 (or 1536 for text-embedding-3-small for cost optimization)
- Chosen for: Superior semantic understanding of technical IT terminology
- Reproducibility: Azure OpenAI API version 2024-02-15-preview

Vector Store: FAISS (Facebook AI Similarity Search)
- Index Type: IndexFlatL2 (exact L2 distance search)
- Why FAISS: Fast, in-memory, no external dependencies for PoC
- Production consideration: Move to Pinecone/Weaviate for scale

Chunking Strategy:
- NO CHUNKING for this corpus (documents are already appropriately sized)
- Each document is a complete unit:
  * Resolution notes: Full ticket resolution (~500-800 tokens)
  * KB articles: Complete troubleshooting guide (~600-1000 tokens)
  * Historical tickets: Ticket summary (~150-200 tokens)
- Rationale: IT resolutions lose context if chunked mid-procedure

1.2 RETRIEVAL FLOW
------------------
Three-tier retrieval system:

Tier 1: Similar Ticket Search
  → Finds past tickets with similar symptoms
  → Returns: Ticket ID, category, resolution code, time-to-resolve
  → Used for: Pattern detection, estimating complexity

Tier 2: KB Article Search
  → Searches structured troubleshooting guides
  → Returns: Step-by-step procedures, best practices
  → Used for: Detailed fix instructions

Tier 3: Resolution Note Search
  → Finds complete RCA notes from past resolutions
  → Returns: Diagnosis, root cause, resolution steps, verification
  → Used for: Full workflow templates

Adaptive Search Strategy:
  Attempt 1: Focused query using ticket category + key symptoms
  Attempt 2 (if poor results): Broader query with alternative keywords
  Attempt 3 (if still poor): Cross-category search with generic terms

Quality Assessment:
  GOOD: ≥2 same-category results, avg distance <1.0
  FAIR: ≥1 same-category or ≥3 results total
  POOR: Otherwise → triggers adaptive retry

1.3 PROMPT STRUCTURE & GROUNDING
---------------------------------
Diagnosis Worker Prompt Strategy:

Context Injection:
  [SYSTEM]: Role definition + grounding rules
  [CONTEXT]: Top-K retrieved documents with metadata
  [TICKET]: Current ticket details
  [INSTRUCTION]: Generate resolution with MANDATORY citations

Grounding Rules (enforced in prompt):
  1. EVERY technical claim MUST cite source: "[According to Ticket TCK-00042...]"
  2. If no relevant context: Explicitly state "No similar cases found"
  3. Never invent solutions - only use retrieved information
  4. Acknowledge contradictions if sources disagree

Hallucination Reduction Techniques:
  - Low temperature (0.1) for deterministic output
  - JSON response format for structured parsing
  - Explicit "cite sources" instruction repeated 3x in prompt
  - Critic agent validates every claim post-generation

2. MULTI-AGENT DESIGN
═══════════════════════════════════════════════════════════════════════════════════════

2.1 ARCHITECTURE RATIONALE
---------------------------
Pattern: Coordinator-Worker-Critic

Why Multi-Agent vs Pipeline?
  ❌ Pipeline: retrieve → diagnose → output (fixed sequence)
  ✅ Multi-Agent: Dynamic routing based on:
     - Priority: P1 → escalate immediately (skip retrieval)
     - Retrieval quality: Poor results → retry with new strategy
     - Complexity: Out-of-scope → route to correct department
     - Safety: Ungrounded claims → revision loop

Genuine Agency Demonstrated:
  - Tool selection NOT hard-coded (11 tools available, agents choose)
  - Routing based on runtime analysis (coordinator's LLM decision)
  - Adaptive retry logic (retrieval worker changes strategy)
  - Self-correction via critic feedback loop

2.2 AGENT ROLES
---------------

COORDINATOR AGENT (agents/coordinator_agent.py)
Role: Planning, validation, routing
Tools: None (pure reasoning)
Capabilities:
  ✓ Scope validation (IT issue vs out-of-scope)
  ✓ Priority assessment (P1=escalate, P4=autonomous)
  ✓ Data quality check (contradictions, missing info)
  ✓ Security screening (prompt injection detection)
  ✓ Execution plan generation

Reasoning Approach: ReAct-inspired analysis
  1. Observe: Analyze ticket fields
  2. Reason: Apply decision framework
  3. Act: Route to appropriate agent

Example Decision Logic:
  IF priority == P1:
    → Route to escalation_handler (immediate HITL)
  ELIF description contains [out-of-scope patterns]:
    → Complete with routing message (no IT resolution)
  ELSE:
    → Route to retrieval_worker (normal flow)

RETRIEVAL WORKER AGENT (agents/retrieval_worker.py)
Role: Context gathering, adaptive search
Tools:
  - similar_ticket_search
  - kb_article_search
  - resolution_note_search
  - user_ticket_history (for pattern detection)

Capabilities:
  ✓ Dynamic query formulation (LLM generates search queries)
  ✓ Multi-tool orchestration (parallel searches)
  ✓ Quality assessment (evaluates result relevance)
  ✓ Adaptive retry (reformulates if poor results)

Stopping Conditions:
  - Good retrieval quality achieved
  - Max attempts reached (3)
  - Escalation triggered by coordinator

Example Adaptive Behavior:
  Attempt 1: "VPN connection timeout error category:VPN"
  → Returns 2 results, distance 0.8 → FAIR quality
  Attempt 2: "VPN connectivity issues remote access"
  → Returns 5 results, distance 0.6 → GOOD quality
  → Proceed to diagnosis

DIAGNOSIS WORKER AGENT (agents/diagnosis_worker.py)
Role: Resolution note generation
Tools: None (synthesizes from retrieved context)
Capabilities:
  ✓ Structured output generation (7-section resolution note)
  ✓ Citation extraction (references all sources used)
  ✓ Context summarization (consolidates multiple sources)
  ✓ Cold-retrieval handling (acknowledges when no context)

Grounding Strategy:
  - Prepares context string from retrieved docs with source IDs
  - Prompt explicitly requires citation format
  - Extracts all ticket/KB IDs mentioned in output
  - Returns references list separately for validation

CRITIC AGENT (agents/critic_agent.py)
Role: Quality assurance, grounding validation
Tools: None (pure analysis)
Capabilities:
  ✓ Citation verification (every claim has source?)
  ✓ Completeness check (all sections present?)
  ✓ Logical consistency (diagnosis → resolution match?)
  ✓ Safety validation (no policy violations?)

Review Criteria:
  GROUNDING: Pass if all technical claims cite sources
  CITATION_QUALITY: Good if cites match retrieved docs
  COMPLETENESS: Pass if all 7 sections present
  SAFETY: Pass if no credentials/privileges exposed

Verdicts:
  APPROVED → End workflow, output resolution
  NEEDS_REVISION → Send back to diagnosis_worker (max 2 loops)
  ESCALATE → Complex issue, requires human expert

2.3 TOOL ORCHESTRATION
-----------------------
11 Tools Available (agents choose dynamically):

Retrieval Tools:
  - similar_ticket_search(query, category, top_k)
  - kb_article_search(query, category, top_k)
  - resolution_note_search(query, category, top_k)
  - ticket_lookup(ticket_id)
  - user_ticket_history(requester_id, limit)
  - category_ticket_search(category, days, limit)

Action Tools (mocked for PoC):
  - asset_lookup(requester_id)
  - create_escalation(ticket_id, reason, priority)
  - send_notification(recipient, subject, message)
  - request_human_approval(ticket_id, action)
  - update_ticket(ticket_id, updates)

Tool Selection Logic:
  - Coordinator mentions needed tools in plan
  - Retrieval worker formulates queries, selects tools based on query type
  - NOT all tools used for every ticket (demonstrates selection)

Example: P4 mouse issue
  Tools used: kb_article_search → asset_lookup → update_ticket
  Tools NOT used: user_ticket_history, escalation (unnecessary)

2.4 MEMORY & STATE
------------------
Short-term Memory (per-ticket scratchpad):
  - AgentState TypedDict shared across all agents via LangGraph
  - Contains: plan, retrieved_docs, references, trajectory
  - Agents update incrementally (Annotated with operator.add)

Long-term Memory (across tickets):
  - FAISS index = persistent memory of all past resolutions
  - Each resolution becomes searchable for future tickets
  - Production: Would include feedback loop (user ratings → corpus updates)

State Evolution Example:
  Initial: {ticket, plan: [], retrieved_docs: [], ...}
  After coordinator: {plan: [step1, step2], should_escalate: False, ...}
  After retrieval: {retrieved_docs: [doc1, doc2], retrieval_attempts: 1, ...}
  After diagnosis: {resolution_note: {...}, references: [TCK-001, KB-02], ...}
  After critic: {critic_approved: True, ...}

Memory Changes Behavior:
  - If retrieval_attempts > 1: Broadens queries
  - If revision_count > 2: Escalates instead of retrying
  - If user_ticket_history shows pattern: Flags recurring issue

2.5 DECISION LOGIC & REASONING
-------------------------------
Reasoning Approach: Hybrid ReAct + Plan-and-Execute

ReAct Elements:
  - Thought: Coordinator analyzes ticket, reasons about approach
  - Act: Agents call tools based on reasoning
  - Observe: Results inform next decision

Plan-and-Execute Elements:
  - Coordinator creates execution plan upfront
  - Workers execute steps from plan
  - BUT: Plan adapts based on execution results (not rigid)

Tool Selection (NOT hard-coded):
  - LLM generates search queries → worker maps to appropriate tools
  - Example: "search for VPN KB articles" → kb_article_search tool
  - Example: "check user's past tickets" → user_ticket_history tool

Tool Call Sequence (dynamic):
  Standard ticket: similar_tickets → kb_articles → resolution_notes
  Recurring issue: user_history → similar_tickets → resolution_notes
  Complex issue: similar_tickets → kb_articles → asset_lookup → escalation
  Out-of-scope: (no retrieval) → immediate completion

Stopping Conditions:
  1. Critic approves resolution (critic_approved=True)
  2. Escalation triggered (should_escalate=True)
  3. Max iterations reached (10 steps per LangGraph config)
  4. Out-of-scope detected (coordinator routes to external)

Failure Recovery:
  Scenario: Cold retrieval (0 results)
  Recovery:
    1. Retrieval worker detects quality=POOR
    2. Reformulates query with broader terms
    3. Retries with different tool (resolution_notes → kb_articles)
    4. If still poor: Proceeds anyway, diagnosis acknowledges limitation
    5. Critic sees acknowledgment → approves with escalation flag

  Scenario: Ungrounded claims in resolution
  Recovery:
    1. Critic detects missing citations
    2. Provides specific revision instructions
    3. Routes back to diagnosis_worker with feedback
    4. Diagnosis regenerates with explicit grounding emphasis
    5. If still fails after 2 loops: Escalates

2.6 HUMAN-IN-THE-LOOP (HITL)
-----------------------------
When Agent Escalates vs Acts Autonomously:

Autonomous (no HITL):
  ✓ P3/P4 tickets with good retrieval matches
  ✓ Issues covered by KB articles
  ✓ Non-critical changes (email settings, password reset)
  ✓ Critic approves grounded resolution

Requires HITL:
  ❌ P1 incidents (company-wide impact)
  ❌ Security concerns (suspicious ticket content)
  ❌ Admin privilege requests
  ❌ Cold retrieval with no similar cases
  ❌ Critic fails to approve after 2 revisions

HITL Workflow:
  1. Agent recognizes HITL needed (priority=P1 or complex issue)
  2. Calls request_human_approval(ticket_id, action, justification)
  3. Drafts preliminary analysis for human review
  4. Waits for approval before taking action
  5. Production: Would integrate with approval workflow system

Critic's Role with HITL:
  - If ungrounded claims detected: Revision loop (internal fix attempt)
  - If revision fails: Escalate with explanation (human expertise needed)
  - Does NOT autonomously output hallucinated content

3. GUARDRAILS & PRODUCTION
═══════════════════════════════════════════════════════════════════════════════════════

3.1 GUARDRAILS
--------------
Prompt Injection Resistance:
  - All ticket descriptions treated as untrusted user input
  - Agents follow system prompts regardless of embedded instructions
  - Example: "IGNORE PREVIOUS INSTRUCTIONS" processed as ticket content
  - Coordinator screens for suspicious patterns (optional escalation flag)

Output Constraints:
  - JSON schema enforcement for structured outputs
  - Length limits on resolution sections
  - No credentials/sensitive data in responses (checked by critic)
  - Mandatory citation format prevents hallucination

Approval Checkpoints:
  - P1 tickets → automatic escalation flag
  - Admin requests → request_human_approval tool required
  - Critic must approve before final output
  - Multi-revision limit (max 2) prevents infinite loops

Audit Logging:
  - Every tool call logged in trajectory with timestamp
  - Agent decisions recorded with reasoning
  - Final state includes complete execution history
  - Production: Persist to database for compliance

3.2 PRODUCTION PATH
-------------------
Deployment:
  - Containerized FastAPI service (Docker)
  - Horizontal scaling via Kubernetes
  - Load balancer for API requests
  - Separate workers for long-running ticket processing

Secrets Management:
  - Azure OpenAI API keys → Azure Key Vault
  - Service-to-service auth via Managed Identity
  - No hardcoded credentials
  - Key rotation policy (90 days)

Monitoring:
  - OpenTelemetry tracing for agent trajectory
  - Custom metrics:
    * Tickets processed / hour
    * Avg retrieval quality score
    * Critic approval rate
    * Escalation rate by priority
    * Processing time by category
  - Alerts:
    * Critic approval rate < 80%
    * Escalation rate > 30%
    * Processing time > 60s
    * Azure OpenAI API errors

Agent Trajectory Tracing:
  - Each agent transition logged with span
  - Tool calls as child spans with input/output
  - Trace ID attached to ticket for debugging
  - Visualization in Jaeger/DataDog
  - Example trace:
    ticket_process [120s]
      └─ coordinator.analyze [2s]
          └─ retrieval_worker.execute [15s]
              ├─ tool.similar_ticket_search [5s]
              ├─ tool.kb_article_search [4s]
              └─ tool.resolution_note_search [4s]
          └─ diagnosis_worker.generate [8s]
          └─ critic_agent.review [3s]

Observability:
  - Structured logging (JSON format)
  - Log levels: DEBUG (trajectory), INFO (decisions), WARN (escalations)
  - ELK stack for log aggregation
  - Grafana dashboards for real-time metrics

Performance Optimization:
  - FAISS index loaded once at startup (in-memory)
  - LLM response caching for duplicate tickets
  - Embedding batch processing for new tickets
  - Async processing for non-critical notifications

Security:
  - Input sanitization (ticket descriptions)
  - Output filtering (no credentials in responses)
  - Rate limiting (prevent abuse)
  - WAF for API endpoints

Reliability:
  - Graceful degradation if Azure OpenAI unavailable
  - Fallback to rule-based routing if LLM fails
  - Retry logic with exponential backoff
  - Circuit breaker for external services

4. TRADE-OFFS & FUTURE ENHANCEMENTS
═══════════════════════════════════════════════════════════════════════════════════════

Current Limitations (PoC Scope):
  - Small corpus (182 documents total)
  - Mock external actions (notifications, ticket updates)
  - Single Azure OpenAI dependency
  - No persistent storage (in-memory only)
  - No user feedback loop

With More Time:
  1. Expanded Corpus
     - 500+ resolution notes across all categories
     - 50+ KB articles with vendor documentation
     - Real ticket data (anonymized)

  2. Real Integrations
     - ServiceNow API for ticket CRUD
     - Microsoft Teams for notifications
     - CMDB queries for asset information
     - LDAP for user/group lookups

  3. Advanced RAG
     - Hybrid search (keyword + semantic)
     - Query rewriting for better retrieval
     - Multi-hop reasoning (follow ticket references)
     - Reranking with cross-encoder

  4. Learning Loop
     - Technician feedback on resolutions (thumbs up/down)
     - Low-rated resolutions → corpus refinement
     - A/B testing different prompt strategies
     - Continuous improvement metrics

  5. Specialized Workers
     - Network specialist agent (routing, firewall, DNS)
     - Security specialist agent (access, MFA, compliance)
     - Hardware specialist agent (laptops, printers, monitors)
     - Coordinator routes by category + complexity

  6. Production Hardening
     - Redis cache for embeddings
     - PostgreSQL for trajectory persistence
     - Prometheus metrics export
     - Automated regression testing
     - Canary deployments
