1. DATA GENERATION ASSUMPTIONS
-------------------------------

Ticket Distribution:
✓ Most tickets are P3/P4 (~85%) - routine issues resolved quickly
✓ VPN and Email are highest volume categories (~45% combined)
✓ ~5% of tickets escalate to formal cases
✓ Resolution rate: 75% Fixed, 15% Workaround, 5% Escalated, 5% Other

Realistic Patterns:
✓ P1 tickets more likely to breach SLA (time-sensitive)
✓ Workaround resolutions more likely to be reopened (~25% vs 5% baseline)
✓ Lower CSAT scores correlated with reopened tickets and SLA breaches
✓ Recurring issues create clusters (same user reporting VPN issues multiple times)

Ticket Relationships:
✓ case_id links escalated tickets to RCA/problem records
✓ Only ~3% of tickets have case_id (realistic escalation rate)
✓ Multiple tickets can reference same case_id (duplicate issue tracking)

Knowledge Corpus:
✓ 25 resolution notes cover major categories proportionally
✓ 7 KB articles provide depth on common issues (VPN, Email, Account, Hardware, etc.)
✓ Resolution notes follow company template structure consistently
✓ KB articles include step-by-step procedures, prerequisites, known issues

Held-Out Split:
✓ Test scenarios are COMPLETELY separate from training corpus
✓ No overlap between test ticket IDs and indexed documents
✓ Retrieval quality measured honestly (agent hasn't "seen" test tickets)

2. TECHNICAL ASSUMPTIONS
-------------------------

LLM Capabilities:
✓ Azure OpenAI GPT-4 has sufficient reasoning for coordination & analysis
✓ Temperature 0.1 provides deterministic output without eliminating reasoning
✓ JSON mode reliably produces parseable structured outputs
✓ Context window (128K tokens) sufficient for ticket + retrieved docs

Embedding Quality:
✓ text-embedding-3-large captures semantic similarity of IT terminology
✓ FAISS L2 distance correlates with relevance for this domain
✓ Top-5 retrieval provides enough context without overwhelming the LLM
✓ No need for domain-specific fine-tuned embeddings at this scale

Agent Coordination:
✓ LangGraph StateGraph correctly manages shared state across agents
✓ Conditional edges reliably route based on state values
✓ No race conditions (single-threaded execution per ticket)
✓ State updates via Annotated[List, operator.add] accumulate correctly

Tool Reliability:
✓ Mock tools (escalation, notification) simulate realistic success/failure
✓ FAISS search always returns results (even if low quality)
✓ No network failures in Azure OpenAI API calls (production: add retry)
✓ Ticket lookup never fails (all ticket IDs valid in dataset)

4. EVALUATION ASSUMPTIONS
--------------------------

Success Criteria:
✓ Grounding: 100% of technical claims should cite sources (measured by critic)
✓ Tool selection: Agent uses ≥2 different tools per ticket (demonstrates choice)
✓ Escalation: P1 always escalates, P4 never escalates
✓ Cold retrieval: Agent acknowledges limitations and retries with broader queries
✓ Adversarial: Agent ignores malicious instructions, focuses on real issue

Metrics:
✓ Overall score >0.8 across all scenarios = success
✓ Individual metric >0.7 = pass for that dimension
✓ Guardrail effectiveness 1.0 = critical (security)
✓ Processing time <60s acceptable for PoC

Human Baseline:
✓ Assumed human technician would escalate P1 immediately
✓ Assumed human would cite KB articles when available
✓ Assumed human would acknowledge when unsure (cold retrieval)
✓ Agent meets/exceeds human baseline on grounding (cites more consistently)

   Current: Single retrieval → generate resolution
   Target: Follow chains of related tickets
   
   Example:
   Ticket: "VPN not working"
   → Retrieves: TCK-00042 mentions "fixed by update in CASE-00015"

10. Cost Optimization
    Current: No caching, all LLM calls fresh
    Target: 50% cost reduction via caching & optimization
    
    Strategies:
    - Embedding cache: Hash ticket description, reuse embeddings
    - Resolution cache: If ticket very similar (distance <0.1), reuse
    - Prompt optimization: Reduce token count 20% via compression
    - Model selection: GPT-3.5 for simple tickets, GPT-4 for complex
    
    Implementation:
    - Redis for embedding & resolution caches (TTL: 7 days)
    - Similarity check before processing
    - Coordinator classifies complexity → routes to appropriate model
    - Monitor cache hit rate (target: 30%)
    
    Cost Analysis (estimate):
    - Before: $0.50/ticket (embeddings + 3 LLM calls)
    - After: $0.25/ticket (30% cache hits, GPT-3.5 for 50% of tickets)
    - At 10K tickets/day: Save $2,500/day = $75K/month
    
    Impact: Cost reduction enables broader deployment

11. Self-Healing Infrastructure
    Vision: Agent doesn't just draft resolutions, it executes them
    
    Safe Actions Agent Can Execute:
    - Password resets (after MFA verification)
    - Group membership adds (with manager approval)
    - Service restarts (after impact analysis)
    - Configuration resets (with rollback plan)
    
    Safety Rails:
    - Dry-run mode: Shows what would be done, requires approval
    - Rollback capability: Every action logged with undo script
    - Blast radius limit: Max 1 user affected per action
    - Escalation: If uncertainty >20%, require human approval
    
    Example Flow:
    Ticket: "Account locked"
    Agent: Checks last lockout, verifies identity, unlocks account
    User: Can login immediately (no human in loop)
    Technician: Reviews action log post-facto
    
    Impact: True automation, not just assistance

RISKS & MITIGATIONS
═══════════════════════════════════════════════════════════════════════════════════════

Risk 1: Agent hallucinations lead to incorrect resolutions
Mitigation:
  - Critic agent validates all claims have citations
  - Low temperature (0.1) for deterministic output
  - Human review of low-confidence resolutions
  - Feedback loop improves over time

Risk 2: Azure OpenAI API outage breaks entire system
Mitigation:
  - Fallback to rule-based routing (if category=VPN → KB-0001)
  - Queue tickets during outage, process when service restored
  - Future: Add local LLM fallback (Llama 3.1 70B)

Risk 3: Adversarial users exploit prompt injection
Mitigation:
  - All ticket descriptions treated as untrusted input
  - System prompts not overridable by user content
  - Coordinator screens for suspicious patterns
  - Security audit logging of all attempted injections

Risk 4: Retrieval quality degrades over time (corpus drift)
Mitigation:
  - Monitor retrieval quality score trend
  - Alert if average distance increases >20%
  - Monthly corpus refresh with new resolutions
  - Technician feedback flags poor resolutions

Risk 5: Agent makes decisions beyond its authority
Mitigation:
  - Hard constraints: Cannot execute privileged actions
  - HITL required for P1, security, admin requests
  - Dry-run mode for testing new capabilities
  - Audit log of all agent decisions
