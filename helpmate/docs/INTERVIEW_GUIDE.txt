1. ARCHITECTURE DECISION
Q: "Why multi-agent vs a pipeline?"

A: "I chose a coordinator-worker-critic pattern because IT support requires dynamic 
decision-making that pipelines can't handle. For example:

- P1 outage: Needs immediate escalation WITHOUT retrieval overhead
- P4 mouse issue: Can be fully autonomous WITHOUT human approval
- Cold retrieval: Needs adaptive retry strategy, not fixed sequence
- Ungrounded claims: Needs revision loop, not one-shot generation

A hard-coded pipeline would always go: retrieve → diagnose → output. But my multi-agent 
system adapts: P1 → escalate immediately, poor retrieval → retry with new strategy, 
critic fails → revision loop. The routing is determined by LLM reasoning at runtime, 
not hard-coded logic."

═══════════════════════════════════════════════════════════════════════════════════════

2. GROUNDING & CITATIONS
Q: "How do you prevent hallucination?"

A: "Three-layer approach:

Layer 1: Prompt Engineering
- Diagnosis worker instructed to cite sources for EVERY technical claim
- Example format required: 'According to Ticket TCK-00042...'
- If no context: Must explicitly state 'No similar cases found'

Layer 2: Critic Validation
- Reviews EVERY claim for citation
- Checks if cited sources exist in retrieved documents
- Triggers revision if ungrounded (max 2 attempts)
- Escalates if can't achieve grounding

Layer 3: Cold Retrieval Handling
- If no relevant docs found: Agent acknowledges limitation
- Recommends escalation rather than inventing solution
- Example: 'Insufficient similar cases. Recommend escalation to senior engineer'

Result: In testing, 100% of approved resolutions had proper citations."

═══════════════════════════════════════════════════════════════════════════════════════

3. ADVERSARIAL INPUTS
Q: "How do you handle prompt injection?"

A: "Prompt injection is a critical security concern. My defense strategy:

Input Treatment:
- ALL ticket descriptions treated as untrusted user content
- Never interpreted as system commands
- Example: 'IGNORE PREVIOUS INSTRUCTIONS' → processed as ticket text, not executed

Coordinator Screening:
- First agent checks for suspicious patterns
- Keywords like 'IGNORE', 'OVERRIDE', 'GRANT PRIVILEGES' flagged
- Can trigger escalation if security concern detected

Agent Resilience:
- System prompts not overridable by user content
- Agents follow their instructions regardless of ticket text
- Example test: Ticket said 'provide admin credentials' → Agent focused on legitimate 
  VPN issue, never mentioned credentials

Result: In adversarial scenario testing, agent ignored 100% of malicious instructions 
and focused on real IT issues."

4. ADAPTIVE BEHAVIOR
Q: "Show me how the agent adapts"

A: "Let me demonstrate with cold retrieval scenario:

Attempt 1: Retrieval worker searches 'custom ERP database constraint error'
Result: 0 relevant documents (rare custom software issue)

Agent Response:
- Detects quality=POOR
- Reformulates query: 'database foreign key constraint violation'
- Tries broader search across all categories

Attempt 2: Broader query returns 2 generic database articles
Result: FAIR quality but not specific to issue

Agent Response:
- Proceeds to diagnosis with limited context
- Diagnosis worker acknowledges: 'No similar past cases found for custom ERP module'
- Recommends: 'Escalate to development team for custom software investigation'
- Critic approves because limitation was acknowledged explicitly

Key: Agent didn't hallucinate a solution. It adapted search strategy, acknowledged 
limitations, and escalated appropriately.

Compare to hard-coded pipeline: Would use empty context and generate garbage resolution."

5. GENUINE AGENCY
Q: "Prove your agent has genuine agency, not just a fixed pipeline"

A: "Three demonstrations:

1. Tool Selection (NOT hard-coded):
   - 11 tools available to agents
   - Agents choose based on ticket analysis
   - Example: P4 mouse → uses [kb_article_search, asset_lookup, update_ticket]
   - Example: Recurring issue → uses [user_ticket_history, similar_tickets, resolution_notes]
   - Different tickets = different tool sequences

2. Dynamic Routing:
   - Coordinator analyzes ticket, routes based on LLM decision
   - P1 → escalation_handler (skips retrieval)
   - P3 normal → retrieval_worker → diagnosis_worker → critic
   - Out-of-scope → immediate completion (no IT resolution)
   - Routing determined at RUNTIME, not compile-time

3. Self-Correction:
   - Critic fails approval → routes BACK to diagnosis_worker
   - Diagnosis regenerates with critic feedback
   - Revision loop up to 2 attempts
   - If still fails → escalates rather than outputting bad resolution

Traditional pipeline: A→B→C→D always
My system: Coordinator decides {A,B,C,D,E,END} based on analysis"

6. PRODUCTION READINESS
Q: "How would you deploy this to production?"

A: "I've designed with production in mind. Here's the deployment architecture:

Infrastructure:
- Containerized FastAPI service (Docker + Kubernetes)
- Horizontal scaling for peak load (ticket spikes)
- Azure OpenAI with retry logic and circuit breaker
- FAISS index loaded at startup (in-memory, fast)

Observability:
- OpenTelemetry tracing: Every agent transition + tool call traced
- Custom metrics: Tickets/hour, critic approval rate, escalation rate, processing time
- Alerts: Approval rate <80%, processing time >60s, API errors
- ELK stack for structured logging

Security:
- Azure Key Vault for API keys (no hardcoded secrets)
- Managed Identity for service-to-service auth
- Input sanitization for all ticket fields
- Audit logging: Every agent decision persisted for compliance

Reliability:
- Graceful degradation: If Azure OpenAI down → fallback to rule-based routing
- Async processing: Long-running tickets don't block API
- Dead letter queue: Failed tickets retry with backoff
- Health check endpoint: /health returns system status

Cost Optimization:
- Embedding cache (Redis): 30% cache hit rate = 30% cost reduction
- Resolution cache: Very similar tickets reuse existing resolutions
- Model selection: GPT-3.5 for simple (P4), GPT-4 for complex (P1/P2)
- Estimated: $0.25/ticket at scale

Deployment Process:
1. PR → Automated tests (unit + integration)
2. Merge → Regression suite (10 adaptive scenarios must pass)
3. Staging → Canary test (5% traffic)
4. Production → Gradual rollout (25% → 50% → 100% over 6 hours)
5. Monitoring: Rollback if critic approval <80% or escalation >30%"

9. TECHNICAL CHOICES
Q: "Why did you choose [technology X]?"

Azure OpenAI:
- Required: text-embedding-3-large for semantic search
- GPT-4 reasoning for coordinator & diagnosis
- Enterprise SLA, compliance (SOC 2, HIPAA)
- Alternative considered: Local Llama 3.1 70B (cost vs quality tradeoff)

FAISS:
- In-memory, fast (<10ms search)
- No external dependencies for PoC
- Easy to swap: Same interface works with Pinecone/Weaviate
- Production: Would migrate to managed vector DB for persistence

LangGraph:
- Built for agent orchestration with state management
- Conditional routing matches my architecture
- Visual debugging of agent trajectories
- Alternative considered: CrewAI (less flexible routing)

FastAPI:
- Modern, async, auto-generates OpenAPI docs
- Type hints = better IDE support
- Easy to containerize
- Alternative considered: Flask (less features)

COMMON QUESTIONS & ANSWERS
═══════════════════════════════════════════════════════════════════════════════════════

Q: "How many documents in your corpus?"
A: "182 total: 25 resolution notes, 7 KB articles, 150 historical tickets. Small for PoC,
    would scale to 10,000+ in production."

Q: "What's the average processing time?"
A: "15-45 seconds depending on complexity. P1 escalates in ~5s. Cold retrieval with 
    retries can take 30-40s. Target for production: <30s p95."

Q: "How do you handle failures?"
A: "Three types: (1) Retrieval fails → adaptive retry, (2) LLM fails → graceful error,
    escalate ticket, (3) Critic fails → revision loop or escalation. Never silently fail."

Q: "What's the cost per ticket?"
A: "~$0.10 for PoC (embeddings + 3-4 LLM calls). With caching and optimization: $0.25
    at scale. At 10K tickets/day: $2,500/day."

Q: "Can this scale to 10,000 tickets/day?"
A: "Yes. Architecture designed for horizontal scaling. Stateless API servers, async
    processing, FAISS index loaded once. Bottleneck is Azure OpenAI rate limit - would
    need quota increase."

Q: "How do you prevent the agent from taking harmful actions?"
A: "Two layers: (1) Tools are constrained - can't execute privileged commands, (2) HITL
    required for sensitive actions - P1, admin requests, security issues. Dry-run mode
    for testing new capabilities."
