ARCHITECTURE HIGHLIGHTS
═══════════════════════════════════════════════════════════════════════════════════════

Multi-Agent Design:

  ┌─────────────────┐
  │  Coordinator    │ ← Analyzes ticket, creates plan, routes
  └────────┬────────┘
           │
      ┌────┴────┐
      ▼         ▼
  ┌──────────┐ ┌──────────────┐
  │Retrieval │ │  Diagnosis   │ ← Workers execute with tools
  │  Worker  │ │   Worker     │
  └────┬─────┘ └──────┬───────┘
       │              │
       ▼              ▼
  ┌──────────────────────┐
  │    Critic Agent      │ ← Validates grounding, triggers revisions
  └──────────────────────┘

Why Multi-Agent?
- Dynamic routing: P1→escalate, P3→autonomous
- Adaptive behavior: Cold retrieval→retry
- Self-correction: Ungrounded→revision loop
- NOT a pipeline: Tool selection & routing at runtime

11 Tools Available:
1. similar_ticket_search      7. asset_lookup
2. kb_article_search          8. create_escalation
3. resolution_note_search     9. send_notification
4. ticket_lookup             10. request_human_approval
5. user_ticket_history       11. update_ticket
6. category_ticket_search

Agents choose dynamically based on ticket analysis (not hard-coded sequence).

KEY TALKING POINTS
═══════════════════════════════════════════════════════════════════════════════════════

Why Multi-Agent?
"IT support requires dynamic decision-making. A P1 outage needs immediate escalation
WITHOUT retrieval overhead. A P4 mouse issue can be fully autonomous WITHOUT human
approval. A hard-coded pipeline can't adapt to these different workflows. My multi-agent
system routes based on LLM reasoning at runtime, not compile-time logic."

How Grounding Works:
"Three layers: (1) Prompt engineering - diagnosis worker instructed to cite sources,
(2) Critic validation - checks every claim has citation, (3) Cold retrieval handling -
if no context, explicitly acknowledge and escalate. Result: 100% of approved resolutions
have proper citations."

Adversarial Defense:
"All ticket descriptions treated as untrusted user input. Agents follow system prompts
regardless of embedded instructions. Coordinator screens for suspicious patterns. In
testing, agent ignored 100% of malicious commands and focused on real IT issues."

Production Readiness:
"Designed for horizontal scaling: containerized FastAPI, async processing, in-memory
FAISS loaded once. Observability: OpenTelemetry tracing, custom metrics, alerts.
Security: Key Vault for secrets, input sanitization, audit logging. Would add: caching
(30% cost reduction), monitoring dashboards, CI/CD pipeline with regression tests."

With More Time:
"Three priorities: (1) Expand corpus to 500+ resolutions, (2) Real ServiceNow integration
for end-to-end automation, (3) Feedback loop where technician ratings improve the agent.
Medium-term: Specialized workers (network, security, hardware agents). Long-term:
Self-healing capabilities with safe action execution."
